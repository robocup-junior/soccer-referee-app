# DESIGN — Issue #82: iOS MAC handling (connection-id vs hardware-MAC split)

> Status: **DESIGN / not yet implemented.** Needs a real iPhone to verify (owner
> rule: device-test before push). Written 2026-07-02.
> Fixes #82 (auto "MAC loading" not working on iOS) and removes the RAVF001 iOS
> limitation carried knowingly by PR #91 / issue #85.

---

## 1. Problem

A `Module` today has a single `macAddress` field (`lib/models/module.dart:68`)
that is used for **two different purposes at once**:

1. **Connection identity** — the string passed to `BluetoothDevice.fromId(...)`
   to open the BLE link (`applyPresetConfig`, `module.dart:713`).
2. **Hardware MAC** — the module's permanent, globally-unique address, which the
   scoreboard server stores and matches against (`home_module_macs`, #70).

On **Android** these are the *same string* (a BLE peripheral is addressed by its
hardware MAC), so the conflation is invisible and everything works.

On **iOS** they are **different**. Apple's CoreBluetooth hides the hardware MAC
for privacy and never lets you connect by it. Instead iOS assigns each peripheral
a **UUID** that is:

- generated by *this* iPhone (not the module),
- **different on every phone** (phone A sees module X as UUID-1, phone B as UUID-2),
- may change on app reinstall / re-pair.

`Module.setBleDevice` stores `bleDevice!.remoteId.toString()` into `macAddress`
(`module.dart:621`) — a **UUID on iOS**. Consequences:

- **#82 (auto MAC-load broken on iOS):** the server sends real MACs;
  `applyPresetConfig` calls `BluetoothDevice.fromId(mac)`. On iOS you cannot
  `fromId` a MAC — it needs the UUID — so auto-pairing silently fails.
- **#85 / RAVF001 (reconciliation broken on iOS):** the end-of-match result POST
  reports `macAddress`, i.e. a UUID on iOS, which the server cannot diff against
  its stored MACs. The feature's exact target case — a referee re-pairing a spare
  mid-match on an iPhone — posts something unreconcilable.

**Both bugs share one root cause: the fused field.** Splitting it removes the
defect rather than patching around it.

## 2. The rescue: the MAC *is* recoverable on iOS

The RCJ firmware advertises the real MAC inside the module's BLE **name**. The
active code builds `'RCJs-m_$mac'.toUpperCase()` and matches it against
`platformName` to find a device (`lib/utils/ble_address.dart:51`,
`resolveIosDeviceUuid`). So on iOS the app **can** know the hardware MAC:

- from the **QR scan input** (`handleIosResult` already receives the scanned MAC,
  then discards it — `module_settings.dart:499`), and/or
- by **parsing it back out of `platformName`** once the device is seen/connected.

> ⚠️ Name-format discrepancy to resolve during implementation: active code uses
> the prefix `RCJs-m_` (`ble_address.dart`), but a stale comment says
> `RCJ-soccer_module-XX:XX:…` (`module_settings.dart:483`). The parser MUST match
> the **real firmware advertised name** — confirm the exact prefix against the FW
> repo (`robocup-junior/soccer-communication-module`) before relying on it.

## 3. Design — two named fields

Split the one fused field into two with distinct, honest names:

```
Module {
  connectionId   // how THIS phone connects: UUID on iOS, MAC on Android
  hardwareMac    // the module's real MAC — stable, global, server-known ('' if unknown)
}
```

Semantics:

| Platform | `connectionId` | `hardwareMac` |
|---|---|---|
| Android | the MAC | the same MAC (equal to `connectionId`) |
| iOS | the CoreBluetooth UUID | the real MAC from QR / advertised name (`''` if never known) |

**Naming note:** rather than add a field beside a misleadingly-named one, prefer
renaming the existing `macAddress` → `connectionId` (it *is* the connection
identity everywhere it's used) and adding `hardwareMac`. This keeps every
`fromId(...)` / reconnect call reading a correctly-named field. All ~call sites of
`macAddress` (module.dart, module_settings.dart, cold-resume snapshot) are updated
in one mechanical pass. (If the rename proves too broad, the fallback is to keep
`macAddress` as the connection id and add `hardwareMac` — but the rename is
cleaner and is the recommendation.)

### Populating `hardwareMac`

- **Android:** set `hardwareMac = connectionId` wherever `connectionId` is set
  (they're identical).
- **iOS — QR scan:** in `handleIosResult`, keep the scanned MAC and store it as
  `hardwareMac` alongside the resolved UUID (currently thrown away).
- **iOS — any connected/seen device:** parse the MAC out of `platformName`
  (`RCJs-m_<MAC>` → `<MAC>`) via a small pure helper in `ble_address.dart`, e.g.
  `String? macFromAdvertisedName(String platformName)`. Use it in `setBleDevice`
  and in the scan-result list so a device connected without a QR scan still gets
  its MAC.
- **iOS — manual entry:** today labeled "Enter device UUID" (`module_settings.dart:343`)
  — the user types a UUID, so no MAC is available → `hardwareMac = ''`. Acceptable
  edge case (rare). Optional future nicety: also allow typing a MAC on iOS and
  resolve it (that path already exists via `resolveIosDeviceUuid`), which would
  populate both — see Open Questions.

### The one clean abstraction both features need

Add a per-platform resolver — a single pure/testable seam:

```
// Android: returns hardwareMac unchanged.
// iOS: scans for the module advertising `hardwareMac` and returns its UUID
//      (wraps the existing resolveIosDeviceUuid), or null if not found.
Future<String?> resolveConnectionId(String hardwareMac)
```

- **#82 (auto MAC-load):** the server MAC → `hardwareMac`; `resolveConnectionId`
  → `connectionId`; connect with `connectionId`. Auto-pair becomes a tidy
  per-platform step instead of an impossible `fromId(MAC)` on iOS.
- **#85 (report actual MAC):** `_actualModulesForTeamId` reports `hardwareMac`
  instead of `macAddress`. Correct on both platforms, and — because it's
  **stored** — it still reports the right MAC for a **disconnected/penalised**
  slot at submit time (the "derive live from the name" alternative can't, since a
  disconnected module has no live advertised name).

## 4. Touch points

- `lib/models/module.dart` — rename `macAddress` → `connectionId`; add
  `hardwareMac`; set both in `setBleDevice`; use `connectionId` in `fromId`
  (`applyPresetConfig`), `bleTX/bleRX` (`remoteId`), `restoreFromSnapshot`.
- `lib/utils/ble_address.dart` — add `macFromAdvertisedName(...)` and
  `resolveConnectionId(...)` (the latter wraps the existing `resolveIosDeviceUuid`
  on iOS, pass-through on Android). Keep the careful scan lifecycle already there.
- `lib/screens/module_settings.dart` — QR flow (`handleIosResult`) keeps the
  scanned MAC; manual-entry + scan-list paths populate `hardwareMac` where known;
  auto-pair uses `resolveConnectionId`.
- `lib/services/match_state_store.dart` (`ModuleSnapshot`) — persist BOTH
  `connectionId` (for reconnect) and `hardwareMac` (so a **cold-resumed** match
  still reports the real MAC). Schema bump; default `hardwareMac`/rename on read
  for back-compat with pre-split snapshots.
- `lib/models/game.dart` (`_actualModulesForTeamId`) — report `hardwareMac`;
  remove the RAVF001 iOS-limitation comment added by #85.
- **Wire payload: unchanged.** `actual_modules.{home,away}[].mac` stays; it just
  carries a real MAC on iOS now. **No rcj-scoreboard#72 server change needed.**

## 5. Back-compat & migration

- **Cold-resume snapshot:** bump `ModuleSnapshot` schema; on read, a pre-split
  snapshot has only the old `macAddress` — map it to `connectionId` and leave
  `hardwareMac` empty (on Android it can be back-filled = connectionId; on iOS it
  stays empty until the next connect re-derives it from the name). Never throw.
- **Result outbox / wire:** no change (payload key + shape identical).

## 6. Invariants to respect (from CLAUDE.md)

- **#1 START/STOP latency:** the resolve-scan runs only at *pairing* time, never
  in the START/STOP path. Do not add awaits/queues to `bleSendPlayAll/StopAll`.
- **#5 OS-owned autoConnect:** `resolveConnectionId` must not introduce an
  app-level reconnect loop or re-`connect()` per disconnect. It resolves an
  identity once at pair time; reconnection stays delegated to
  `connect(autoConnect:true)`. The existing `resolveIosDeviceUuid` global scan is
  already written to tear down in `finally` and stop on match — reuse it as-is.
- **#2 double-tap safety / #3 provider tree / #4 portrait:** untouched.

## 7. Testing

- **Unit (no device):** `macFromAdvertisedName` (valid name, wrong prefix, no MAC,
  case-insensitivity); `resolveConnectionId` Android pass-through; `ModuleSnapshot`
  round-trip incl. pre-split back-compat; `_actualModulesForTeamId` reports
  `hardwareMac` for connected + disconnected + never-paired slots.
- **iOS device (required, owner rule):** on a real iPhone — QR-pair a spare, run a
  match, submit; confirm the POST carries the real MAC (not a UUID). Verify auto
  MAC-load (#82): a deep-linked match with `home_module_macs` auto-connects the
  modules on iOS. Confirm cold-resume reconnects and still reports the MAC.
- **Android regression:** existing #85 Android flow (mock + `adb reverse`, see
  `07_SCOREBOARD_DEVICE_TESTING.md`) still posts the real MAC unchanged.

## 8. Open questions

   switch iOS manual entry to accept a **MAC** and resolve it (populating both,
   and matching the Android UX)? The resolve path already exists. Leaning: allow
   MAC entry on iOS too — it removes the last empty-MAC case. Confirm with owner.
2. **Confirm the firmware advertised-name prefix** (`RCJs-m_` vs
   `RCJ-soccer_module-`) against the FW repo before shipping the parser.
3. No server change is required for #85 reconciliation under this design — but
   worth a heads-up to @mrshu on #72 that iOS will start sending real MACs once
   #82 lands (so any UUID-tolerance he may have added server-side is unneeded).
